=== NOTES ===

- The bearer token NEVER appears in any log line (scrubbed above as <TOKEN>; token is
  also never written to any file inside the repo tree).
- The signing audit log line (level INFO, msg "signing: transaction signed successfully")
  contains: request_id, tx_hash, chain_id, nonce. No key material, no tx body, no password.
- The signing audit log line carries the request_id of the INITIALIZE HTTP request
  (b0511fa6 here, the first reqlog line with status 200). The go-sdk's
  StreamableHTTPHandler routes tool calls through the session context established by
  the initialize call, so all tool handlers share that session's request_id. The
  sign_transaction POST's own reqlog line (27ad1c20, status 200, latency 54 ms) has a
  different request_id — that is the transport-layer request, not the tool-handler
  context. Correct correlation: initialize reqlog request_id == signing audit
  request_id (this property is verified by parity_transport_test.go).
- The transaction is NOT broadcast. The binary makes no outbound network calls (ADR-007).
- The WARN lines are expected for the committed keystore fixtures (chmod 644 from git).
  In production: chmod 600 your keystore and password files.
- This transcript is input evidence for the Issue 4.4 end-to-end leak audit.
